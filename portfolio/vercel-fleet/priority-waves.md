# Strategic Priority Waves

## Wave 1 — Studio operating system and core revenue engine

Purpose: establish the factory's own contract, recover the central cloud layer, and document the MAXX delivery stack.

**Batch 1**

1. `vibe-engineering` — central control plane and governance.
2. `pauli-cloud` — broken, intended shared agent add-on and cloud execution layer.
3. `pauli-hermes-agent` — shared agent runtime behind multiple deployments.
4. `macs-agent-portal` — customer-facing MAXX product surface.
5. `maxx-migrations-agentic-systems` — paid migration and operations capability.

## Wave 2 — Active client delivery

Prioritize Asc3nd, Veronika/Kupuri, Macs Digital Media, and other projects with an identifiable buyer, deadline, or upsell path.

## Wave 3 — Owned IP and audience assets

Prioritize Yappyverse, Afromations, Impact City, New World Kids, Pauli Scroll World, and related media systems only where each asset has a distribution or monetization path.

## Wave 4 — Local commercial products

Review Vallarta, construction, transportation, food, pressure washing, marketplace, and local-service projects. Separate active prospects from speculative demos.

## Wave 5 — R&D, forks, templates, and generic deployments

Forks and experiments do not enter Sandcastle until they have one of:

- an active dependency in a revenue project;
- a reusable studio capability with a named owner;
- a validated product hypothesis;
- a retirement or consolidation decision.

## Portfolio decision rules

1. One repository is the unit of factory work, not one Vercel deployment.
2. Duplicate deployments must be consolidated or assigned explicit purposes.
3. `HTTP 200` means reachable, not production-ready.
4. Broken high-leverage infrastructure can outrank healthy low-value demos.
5. Unknown-purpose and generic projects should be archived or renamed before investment.
6. Every active repository needs a commercial or mission outcome, production definition, owner, and next action.
