# Client-Lane Bootstrap Sequence v1

## Decision

Bootstrap client truth sequentially, then execute approved implementation tickets concurrently.

## Priority order

1. Asc3nd — first confirmed paying client; delivery obligation comes first.
2. MACS / MAXX — highest distribution leverage through Stacy and his client network.
3. Kupuri / Synthia — existing internal operating system with a clear higher-value managed-service opportunity.
4. Verified Vallarta — local trust and lead-generation asset requiring collaborative design and editorial work.
5. Lane 5 — selected only after portfolio scoring identifies the strongest revenue-to-effort owned opportunity.

## Required bootstrap gate per lane

Each lane must complete the following before autonomous implementation tickets begin:

1. Client-level master contract.
2. Canonical repository and deployment role map.
3. Read-only repository/system audit.
4. Production and commercial gap list.
5. Founder + digital cofounder review.
6. Approved bounded tickets.

A lane that reaches PRD_READY may begin isolated Sandcastle execution while later lanes are still completing intake and audit.

## Execution model

Ambiguity is resolved sequentially. Execution is parallelized only after the relevant client system has a trusted contract, evidence-backed audit, approved success metric, and bounded ticket.

## Authority boundary

During the first 30 days, agents may work in isolated sandboxes, create branches, previews, tests, evidence and pull requests. They may not autonomously merge protected production branches or publicly deploy client-facing changes.

## Public-facing gate

Technical readiness does not equal public readiness. Public-facing releases require joint founder and digital-cofounder review of design, copy, positioning, content/blog quality, localization, trust, conversion journey and overall presentation before deployment approval.
