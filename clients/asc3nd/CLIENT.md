# Asc3nd — Client Master Contract

## Status

- Client system ID: `asc3nd`
- Commercial state: **PAID CLIENT — ACTIVE DELIVERY OBLIGATION**
- Bootstrap state: `READ_ONLY_AUDIT_IN_PROGRESS`
- Public release authority: `HUMAN_APPROVAL_REQUIRED`

## Purpose

Asc3nd Collective is managed as one client delivery system spanning its public website, interactive strategy workbook, backend/agentic operating system, data persistence, workflows, deployments, and measurable engagement journey.

The unit of management is the **Asc3nd client system**, not any individual repository or Vercel project.

## Locked 30–60 day success outcome

One real acquisition-to-engagement loop must work end to end in production:

`discover Asc3nd → understand the mission → take a meaningful action → data captured correctly → Asc3nd can follow up → outcome measured`

Technical completion is required underneath this outcome but is not itself the success metric.

## Founder/cofounder public-facing gate

No public-facing change is approved for production merely because builds/tests pass.

Before public deployment, Jeremy and the digital cofounder review together:

- positioning and offer clarity;
- visual design and taste;
- copy and mission accuracy;
- English/Spanish parity and localization quality;
- content/blog/editorial quality where applicable;
- mobile experience;
- trust, proof, and credibility;
- calls to action and user journeys;
- accessibility;
- whether the experience is ready to represent Asc3nd publicly.

Required state transition:

`TECHNICALLY_READY → PUBLIC_REVIEW_REQUIRED → PUBLIC_APPROVED → SHIP`

## Authority during first 30 days

Agents may audit, create bounded tickets, work in isolated sandboxes/branches, implement approved tickets, test, verify, open PRs, and produce previews/evidence.

Agents may not autonomously:

- merge protected production branches;
- publicly deploy client-facing changes;
- perform destructive migrations;
- make pricing/legal/customer commitments;
- publish content externally;
- bypass human approval for consequential youth-data, privacy, safety, financial, or public-facing decisions.

## Operating principles

1. Reuse before rebuild.
2. Preserve working repo boundaries unless evidence proves consolidation reduces cost, duplication, failure risk, or founder attention.
3. Read-only audit precedes product-code modification.
4. One bounded ticket at a time per sandbox.
5. Commercial outcome is tracked separately from technical completion.
6. Youth/privacy/safety risks are release blockers until explicitly resolved.
